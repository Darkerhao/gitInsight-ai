# 多角色 AI 日报助手实施计划

## 核心判断：角色的事实来源不同，不是同一份数据的不同表达

上一版把项目经理、产品经理日报当作研发日报的"改写视图"，从 commit 数、影响文件、
研发正文里反推进度和需求——这在编造数据。项目经理和产品经理的日报，事实来源根本不是 Git：

- **研发**：事实来源是 Git（提交、diff、影响文件、仓库时间段）。
- **项目经理**：事实来源是项目推进事实——今日推进事项、里程碑进度、阻塞风险、协调事项、
  会议结论、成员/资源状态、明日安排。
- **产品经理**：事实来源是产品工作事实——需求进展、PRD/原型/评审、用户反馈、验收情况、
  版本规划、待确认问题、协同事项。

因此本方案按"三种角色，三种生成方式"重做，而不是围绕"Git 的表达方式"重做。

## 生成方式

- **研发模式**：沿用 Git 生成流程（`report:generate`）。选择仓库、时间段，AI 汇总提交与 diff。
- **项目经理模式**：展示结构化素材表单（分区 + 补充材料粘贴区），用户填写项目推进素材，
  经 `report:generate-from-material` 由 AI 整理为日报。
- **产品经理模式**：展示产品工作素材表单（分区 + 补充材料粘贴区），用户填写需求/评审/验收/反馈，
  同样经 `report:generate-from-material` 由 AI 整理为日报。

保存、复制、导出、发布飞书对三种角色共用当前选中的正文，无需区分来源。

## 后端契约（IPC）

新增三个通道，遵循项目"四处锁步"约定（main / preload / env.d.ts / shared types）：

- `report:generate-from-material`：入参 `GenerateFromMaterialParams`（角色、日期、汇报人、
  分区素材、补充材料），返回 `MaterialReportResult`。无 AI Key 或调用失败时回退到本地结构化排版，
  并追加 `AI提示：...`，与研发路径的容错策略保持一致。
- `role-material:save` / `role-material:load`：把用户填写的素材按 `(role, date)` 持久化到
  `role_materials` 表，切换角色或次日回来时可恢复草稿。

素材分区定义集中在 `@shared/types` 的 `PROJECT_MANAGER_SECTIONS`、`PRODUCT_MANAGER_SECTIONS`，
前后端共用，避免分区 key 漂移。

## 页面交互

1. 顶部选择日报角色：研发 / 项目经理 / 产品经理。
2. 研发角色：显示仓库选择、日期、提交时间段，走 Git 生成。
3. 项目经理 / 产品经理角色：显示对应素材分区表单 + 补充材料粘贴区，走素材生成。
4. 生成后正文进入可编辑区，按角色分别保存草稿，切换角色不丢失手动修改。
5. 当前角色正文可保存、复制、导出 Markdown 或同步到飞书。

## 风险控制

- 素材角色的日报只基于用户填写的事实，不再从 Git 反推业务结论。
- 无 AI Key 时素材角色回退到结构化排版（按分区罗列用户输入），研发角色回退到基础模板。
- 素材按 `(role, date)` 持久化，重新打开可恢复；研发结果与素材结果互不污染
  （切换到素材角色时清空 `lastReportResult`，避免下游误用 commit 信息）。

## 验收标准

- 顶部可切换三种角色，切换时联动展示对应输入区（仓库范围 / 素材表单）。
- 研发角色沿用 Git 生成，提交记录、影响文件、原始摘要页签正常。
- 项目经理 / 产品经理角色填写素材后可生成日报，未填素材时有明确拦截提示。
- 素材填写后切换角色再返回，内容保留；已生成正文按角色独立保存。
- 保存、复制、导出、发布飞书使用当前选中角色的正文。
- `npm run typecheck`、`npm run build`、`git diff --check` 均通过。

## 后续迭代

- 素材表单支持模板/历史带入，减少每日重复填写。
- 新增 AI 审阅问答，基于日报与素材回答风险与进度问题。
- 打通需求单或项目管理系统，自动带入部分素材，减少手工录入。
- 为团队日报增加多人、多角色聚合能力。
